Character System

300+ characters with 6-second tool loading — from parts assembly to preview and external VRM support
Tool Design, C++ Plugin, UE Integration 2024~2026 Company Project
Unreal Engine 5 C++ Blueprint Data Asset VRM4U Unreal Insights

Problem

As the character count approached 300, we needed a pipeline to systematically manage parts assembly, registration, preview, and loading. Opening the editor pulled in all assets via hard references, requiring a 30-second wait. Checking colors or animations meant launching the game every time. With external VRM support added, manually setting up cameras and thumbnails per model became yet another burden.

Solution

I proposed a VRM save/load prototype using vrm4u early on, which was adopted and is now running in production. I split the work into three systems: an Editor tool, a Runtime character Blueprint, and VRM Reference Points. Each loads Data Assets or external models and lets you verify them against the in-house shader and animations. Artists assemble parts and register presets in the editor, producing Data Assets that the runtime loads asynchronously.

The original character Blueprint had both save and load in a single actor, but the references held by the save side caused a 30-second wait every time the editor opened. By extracting save into a dedicated editor tool and leaving only load in the runtime Blueprint, load time dropped from 30s to 6s. The editor tool also implements the Editor Tick Interface, enabling preview directly on a background without launching the game. For external VRM, Reference Points auto-detects the face via a 4-level fallback and handles camera targeting and thumbnail generation automatically.

System Structure

Three separate codebases, connected through Data Assets and VRM info nodes. The editor tool produces Data Assets. The runtime Blueprint loads them async. VRM Reference Points handles external model info.

Editor — Character Environment Checker

Artists assemble head, hair, and body parts in the editor and save them as presets. The Editor Tick Interface allows verifying characters and animations on a background without launching the game. The runtime reads these Data Assets directly.

Runtime — Character Blueprint

A separate codebase from the editor tool -- a runtime character Blueprint inheriting the base character class. Internal characters load parts asynchronously from editor-produced Data Assets (Head/Hair/Body split), while external VRM loads the entire model through a loader. The same Blueprint handles both, branching only at the load path.

External Model Support — VRM Reference Points

A C++ module written for the VRM4U plugin that updates VRM info nodes in the runtime character Blueprint. It auto-detects the face position of any external VRM/PMX via a 4-level fallback, then uses the result as a camera target to automatically capture 3-angle thumbnails (face close-up, upper body, full body). A commandlet environment enables batch thumbnail generation for all characters without opening the editor. A-pose/T-pose detection is also automatic.

Technical Details

Loading 30s to 6s

Soft Object Reference + Async Loading
Two root causes behind the 30-second load: per-part Hard References for save were all pulled in at editor load, and 80+ unused blend shapes (face morphs) came along too. Extracting save into the editor tool removed all those references, and blend shapes were changed to not preload. Data Assets needed only at load time are processed asynchronously via StreamableManager.RequestAsyncLoad.
Data Asset Only Migration
Inspired by the MetaHuman preset structure. Even with soft references, DataTables must read all rows on first load, making them fundamentally slow. Data Assets only hold what's needed via soft references, so the structure itself is lightweight.
Unreal Insights Profiling
Traced LoadObject call stacks in the CPU Timing and Loading tabs to identify which assets were consuming time. This data provided the evidence needed to strip out Hard References.

Character Actor BP Structure

Unified / Split Skeletal Mesh Loading
Split mode (internal characters) loads Head/Hair/Body parts separately from Data Assets, while unified mode (VRM/PMX) loads the entire model through the VRM4U loader. The same Blueprint handles both with only the load path branching; once loaded, all skeletal mesh components are reset and reattached.
Editor Tick Interface — Editor Preview Without PIE
Implemented the Editor Tick Blueprint Interface in the editor tool, enabling animation playback in the editor environment. Art directors can verify colors, scale, and animations without launching the runtime, so iteration is faster.

Auto Face Detection

VRM Reference Points 4-Level Fallback
Auto-detects face landmarks for external VRM/PMX characters. Level 1: Bone lookup (humanoidBoneTable). Level 2: Material vertices + raycast (Moller-Trumbore). Level 3: Position interpolation. Level 4: Fallback estimation. Detection results are used for camera targeting and automatic thumbnail capture.
